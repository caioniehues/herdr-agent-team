# Wave 5 learnings — seam unification + spawn robustness (2026-07-15)

Coordinator retrospective for #22 (PR #26) and #14+#17 (PR #27). Written per
standing rule: formalize learnings after every task/wave.

## Process learnings

### Central gate must repeat-run threaded test suites
Worker H's suite passed green locally and once centrally, then failed 2/11
repeat runs: `shared_cwd_workers_get_distinct_protocols...` zipped protocol
paths positionally against spec order while parallel launch made
brief-submission order thread-scheduling-dependent. A single gate run samples
one scheduling; the flake only shows under full-suite load. **Rule: any PR
introducing threading gets a 15x `cargo test` loop in the central gate.**
Fix pattern that worked: key assertions by identity (BTreeMap by filename),
force the adversarial order deterministically (condvar `BriefOrder`), never
`--test-threads=1` or sleeps.

### Two-round review with an adversarial re-check pays for itself
Round 1 on PR #27 found 3 MAJORs the tests couldn't see (resume double-launch
into a live TUI; adopted workers wrongly resumable; cross-process run.toml
clobber). Round 2 was briefed adversarially ("does the guard cover crash-after-
CLI-launch AND crash-after-brief?") and confirmed each fix plus found the
residual class (#28, #29) instead of rubber-stamping. **Rule: re-review the fix
commit with per-finding FIXED/NOT-FIXED verdicts, phrased as attack questions.**

### Reviews and all delegation run as herdr pane teammates
An Agent-tool reviewer ran 30 min invisible with no output (second such
incident). Claude pane workers (`pane split` + `pane run claude`) are visible,
steerable, and worked flawlessly twice. Codex and claude both delegate fine
through panes.

### Same-scope partial handoffs: msg-resume, don't respawn
G handed off #22 half-done (production seam yes, fake consolidation no).
One `msg` with the exact remaining scope resumed it in-context to completion.
Cheaper than any respawn; worker kept all its context.

### File follow-up issues at merge time, from review residue
Review residue (unlocked writers elsewhere, stuck-pending adoptee) became
#28/#29 immediately, while context was hot. Nothing relies on memory.

## Mechanics learnings (herdr/plugin operation)

- **Claude TUI paste trap**: a long `pane run` prompt lands as pasted text and
  the Enter is swallowed by paste-mode; agent stays `idle`. Recovery: one empty
  `pane run ""` submits the composer. (This is now also the plugin's own
  submit_verify retry behavior, spec §3.)
- **God pane death recovery**: if the god pane dies mid-run, workers' pointer
  injection and msg targets go stale. Recovery: patch `god_pane_id` in the
  run's `run.toml` to the new pane id; durable inbox reports survive
  regardless — that redundancy (pointer + durable report file) is the design
  working as intended.
- **Interrupted-spawn testing**: `timeout --signal=INT 1.2s <spawn>` reliably
  produces the half-launched state. Post-#17 parallel spawn is so fast (5s
  wall for 2 codex workers, formerly 90s+ each) that interrupts after ~2.5s
  miss entirely.
- **Worktree/workspace hygiene after waves**: plugin `kill` preserves
  worktrees; test/merged worktrees and their local+remote branches need
  explicit removal, and worktree.created plugin hooks leave stray shell
  workspaces (close any workspace whose only pane has `agent: none`).
- **`sd` scope trap**: `sd 'dod5' 'dod5r' spec.toml` also rewrote the brief
  path inside the file. Target replacements precisely or verify after.
- **Monitor discipline**: watch report-file existence/mtime + pane terminal
  states, never scrollback sentinels; every monitor breaks on
  idle/done/blocked/gone so silence can't mask a dead worker.

## Verified product wins (post-#14/#17 behavior)

- 2-worker spawn: 5s wall, both `brief_submitted`/`running`, zero 90s
  agent-info stalls (lazy identity works).
- `spawn --resume`: pending/resources_ready workers complete to
  running/brief_submitted; running workers untouched; second resume is a clean
  no-op ("no spawn-owned pending workers").
- Checkpoint enum (`pending → resources_ready → brief_submitted`) audit trail
  matches reality at every interrupt point tested.
